GitHub

GitHub 就是程序员的新浪微博。

GitHub 不仅仅是用来作为 Git 仓库托管的，更多的是为编程加入了社交元素。

GitHub 与 Git 的区别：

  • Git 是源代码版本管理工具
  • GitHub 是提供远程仓库托管的一个服务平台

使用 GitHub 的前期准备

  • 创建账户
  • 个人设置
  • 使用社区功能
    • Follow
    • Watch

使用 GitHub 进行远程同步

  • 创建仓库
  • clone 已有仓库
  • 本地版本管理
    • add
    • commit
  • push 推送
  • pull 拉取

已有远程仓库

  • clone 到本地
  • 然后在本地提交版本
  • push 到远程仓库

通过 clone 下载下来的仓库，会自动在 remote 中添加一个名字为 origin 的仓库地址。

已有本地仓库

在已有本地仓库（非 clone 的）的情况下，往线上推送。

  1. 在你使用的代码托管平台创建一个远程仓库
  2. git remote add 地址别名 远程仓库地址
  3. git push 地址别名 master

Pull 和 Push

在每次 push 的时候，如果本地仓库的版本和远程仓库版本不一致了，git 是不允许 push 的。

所以，这个时候，就必须在 push 之前先 pull 一下， 拉取最新版本（先确保把本地的版本和远程仓库的版本保持一致）。pull 下来之后，文件与文件之间有的会自动合并，有的会产生冲突（git 也不做这件事儿），所以你找到冲突的文件，找到提交和谁提交的这个版本的文件冲突了的人，两个人一起对比这个文件进行合并。手动合并(删除冲突文件格式)之后再 commit 这个合并之后的版本，然后再 push。

使用 Git 结合 GitHub 多人协作

Pull Request

专业一点应该具有Code Review这个环节，Pull Request 不仅能帮你实现协同，更多的是帮你把Code Review变得更加方便了

其他成员如何协同：

  1. 把源代码仓库 fork 到自己的账户下
  2. 从自己账户下的仓库中clone到自己的本地
  3. 新建分支（功能分支、解决bug分支、性能修补分支）
  4. 基于新分支编写你的代码
  5. 在你的本地新分支功能开发完毕之后，把该分支push到自己fork的那个仓库中
  6. 发起 Pull Request 请求，请求把我的仓库中的这个分支合并到你的某个分支中

仓库的拥有管理者：

  • 收到 Pull Request 合并请求
  • 处理 Pull Request 合并请求
    • 看一下对方的代码是否符合规范，是否有不合理的地方，是否有改错的地方
    • 如果一旦有问题，GitHub 就像普通的聊天工具一样，告诉对方你哪有问题
    • 对方也会收到你的合并反馈
    • 源码提交者和源码管理者可以通过这种方式进行协同交互
    • code view
  • 直到最后源码管理者认为你的代码没有问题，将你提交的代码合并到了我的主仓库中

Collaborators

  • 项目管理者对协同者发出协同邀请
  • 协同者会收到一封协同邀请邮件，在邮件中同意成为该项目的协同者

只要根据上面的操作就可以让一个项目具有多个协同者，提交的时候，所有的协同者都具有直接 push 的权限

关于提交日志中的用户信息的配置问题

  • system
    • 该配置信息存储在git的安装目录中
  • global
    • 该配置信息存储在用户目录下的 .gitconfig 文件中
    • 一般配置自己常用的
  • local
    • 在项目的.git目录中
    • 一般针对特定项目做
